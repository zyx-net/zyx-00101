# 门店交接班异常追踪系统

便利店早中晚班交接班管理 Web 应用，用于登记交接班异常（现金差额、库存短缺），接班人确认，店长复核关闭或退回，全流程操作历史记录和数据持久化到本地 JSON 文件，重启不丢失。对暂时无法当场关闭的异常，可发起"交接整改任务"进行全生命周期跟踪（发起 → 分派 → 处理 → 验收/驳回 → 关闭）。

**新增模块：门店设备巡检与维修闭环** — 支持店长维护设备清单（含CSV批量导入）、巡检模板管理，员工按班次生成巡检单、逐项填写巡检结果，异常项一键转维修单，完整记录报修→接单→完成→验收/退回的状态历史，数据持久化不丢失，权限按门店隔离。

## 技术栈

- 后端：Node.js + Express
- 前端：原生 HTML/CSS/JavaScript（无构建工具）
- 存储：本地 JSON 文件

## 后端模块结构

重构后采用模块化架构，职责清晰：

```
server/
├── middleware/           # 中间件模块
│   ├── auth.js          # 认证与权限（登录验证、店长权限）
│   ├── store.js         # 门店隔离（数据过滤、访问控制）
│   └── conflict.js      # 并发冲突校验（updatedAt 检查）
├── constants/           # 常量定义
│   └── status.js        # 状态常量（班次/异常/任务/设备/巡检/维修）
├── utils/               # 工具函数
│   └── export.js        # 导出工具（CSV/JSON 格式转换）
├── routes/              # 路由模块
│   ├── auth.js          # 认证路由（登录/登出）
│   ├── config.js        # 配置路由（门店/清单/用户）
│   ├── shifts.js        # 班次路由
│   ├── exceptions.js    # 异常路由
│   ├── tasks.js         # 整改任务路由
│   ├── devices.js       # 设备路由
│   ├── inspections.js   # 巡检路由
│   ├── repair-orders.js # 维修单路由
│   ├── export.js        # 导出路由
│   └── history.js       # 操作历史路由
└── store.js             # 数据持久化层（JSON 文件读写）
```

**核心特性：**
- **门店隔离复用**：所有数据列表和导出都通过统一的门店过滤中间件
- **权限规则统一**：店长/员工权限检查集中在 auth.js，避免重复
- **冲突校验标准化**：统一的 updatedAt 冲突检测，409 错误保持原状态不变
- **导出逻辑共享**：CSV/JSON 导出统一处理，权限一致

## 整改任务完整复现流程

对暂时无法当场关闭的异常（需调监控、盘点、联系供应商等），可转为可跟踪的"交接整改任务"。

### 第五步：员工（赵中班）从异常详情发起整改

1. 使用 `staff2 / staff123` 登录
2. 进入"班次管理"→ 班次详情 → 异常列表
3. 在一条"待处理"状态异常行，点击「发起整改」
4. 填写整改表单：
   - 任务标题：可留空自动生成（例如"整改: 现金差额"），也可手动填写
   - 责任人：选择门店内员工（如孙晚班）
   - 截止时间：选择一个日期
   - 处理步骤：例如"1. 重新盘点库存\n2. 调阅收银台监控录像\n3. 联系供应商核对"
   - 附件说明：例如"监控录像片段编号 20260613-007、盘点单 P-0042"
5. 点击「提交」，任务被创建，状态为"待分派"，可在左侧菜单「整改任务」查看

### 第六步：店长（张店长）分派整改任务

1. 使用 `manager1 / manager123` 登录
2. 进入「整改任务」→ 选择范围"待我处理"，即可看到待分派的任务
3. 点击「详情」进入任务详情页
4. 点击「分派任务」，可以变更责任人或沿用指定的责任人，可选填写分派说明
5. 点击「确认分派」，任务状态变为"已分派"，责任人收到待办（筛选"待我处理"即可看到）

### 第七步：责任人（孙晚班）提交整改结果

1. 使用 `staff3 / staff123` 登录
2. 进入「整改任务」→ 范围选"待我处理"，找到分派给我的已分派任务
3. 点击「详情」→ 点击「提交整改完成」
4. 填写处理完成说明：例如"已完成盘点，盘亏 3 瓶可乐，由本人承担；监控录像确认无异常操作，已将货款补齐"
5. 点击「确认提交」，任务状态变为"已提交"，等待店长验收

### 第八步：店长验收关闭或驳回

1. 使用 `manager1 / manager123` 登录
2. 整改任务列表筛选范围"待我处理"或状态"已提交"
3. 点击「详情」进入
4. 如果处理到位：点击「验收关闭」，填写验收意见后确认 → 任务变为"已关闭"
5. 如果仍需整改：点击「驳回」，填写驳回原因（如"未附监控截图，请补充"）→ 任务变为"已驳回"，责任人重新提交

> **驳回循环**：被驳回的任务，责任人修正后可再次提交；店长可再次验收或驳回，直至关闭。每次状态变化都会写入「操作历史」和任务详情中的「状态历史」。

### 整改任务列表与导出

- **独立入口**：左侧导航栏「整改任务」
- **筛选**：按门店、状态、范围（全部/待我处理）筛选
- **"待我处理"的含义**：
  - 店长 = 我门店的待分派任务 + 已提交待验收任务
  - 员工 = 分派给我的已分派/已驳回任务
- **导出**：列表顶部「导出 CSV」/「导出 JSON」按当前筛选条件下载数据

## 整改任务权限与错误提示（不会破坏原状态）

- **非本门店查看**：其他门店员工通过 URL 访问本门店任务详情会报「无权查看非本门店整改任务」
- **跨门店分派/验收/驳回**：非本门店店长执行分派、验收、驳回会报「仅本门店店长可...」
- **非责任人提交**：非责任人且非店长尝试提交会报「仅责任人或店长可提交整改任务」
- **并发修改**：两人同时打开同一个任务提交，后提交者携带旧的 updatedAt 会报「任务已被他人修改，请刷新后重试」（HTTP 409）
- **已关闭任务再次修改**：关闭后再提交验收会报「当前状态 [closed] 不可验收」
- **重复创建任务**：同一异常已有进行中的整改任务时再次发起会报「该异常已有进行中的整改任务，不可重复创建」（HTTP 409）
- **分派给外店人员**：指定责任人不属于本门店时报「责任人必须属于本门店」

# 新增模块：门店设备巡检与维修闭环

## 模块概述

设备巡检与维修闭环完整流程：**店长维护设备清单和巡检模板 → 员工按班次生成巡检单 → 逐项巡检并填写结果 → 异常项转维修单 → 报修→接单→完成→验收/退回**。数据持久化到本地JSON文件，重启不丢失，权限严格按门店隔离。

## 前端独立入口

左侧导航栏新增 4 个菜单：
1. **设备管理** - 设备清单维护、CSV导入、导出

3. **巡检管理** - 巡检单创建、填写、提交、转维修
4. **维修单** - 维修单分派、接单、完成、验收/退回

## 权限控制与错误边界

### 门店隔离
- **外店查看**：海淀店 manager2 登录，访问朝阳店设备/巡检/维修，返回 403「无权查看非本门店...」
- **外店操作**：海淀店 manager2 尝试修改朝阳店设备，返回 403「仅本门店店长可...」

### 角色控制
- **设备/模板维护**：仅店长可新增、修改、删除、导入
- **巡检单提交**：仅巡检人本人或店长可提交
- **维修分派/验收/退回**：仅本门店店长
- **维修完成**：仅接修人或店长

### 并发冲突处理（409）
1. 两个浏览器窗口同时打开同一张维修单详情
2. 窗口A先完成维修（携带较新的 updatedAt）
3. 窗口B后提交，携带旧的 updatedAt → 返回 409「维修单已被他人修改，请刷新后重试」，原状态不变

## 数据持久化文件

新增 4 个持久化文件，均位于 `data/` 目录：

| 文件名 | 内容 |
|---|---|
| `devices.json` | 设备清单 |
| `inspection-templates.json` | 巡检模板 |
| `inspections.json` | 巡检单及巡检项结果 |
| `repair-orders.json` | 维修单及状态历史 |

服务重启后：
- 列表、详情、筛选结果、状态、状态历史 **全部完整保留**
- 导出的 CSV/JSON 内容与重启前一致
- 操作历史记录包含所有新动作（CREATE_DEVICE/IMPORT_DEVICE/CREATE_TEMPLATE/CREATE_INSPECTION/SUBMIT_INSPECTION/CREATE_REPAIR/ASSIGN_REPAIR/COMPLETE_REPAIR/VERIFY_REPAIR/REJECT_REPAIR）

## CSV/JSON 导出

各模块列表顶部均有「导出CSV」「导出JSON」按钮，支持：
- 按当前筛选条件导出（状态、门店、日期等）
- CSV 采用 UTF-8 BOM 编码，Excel 可直接打开
- JSON 格式完整，便于程序解析

## 整改任务数据持久化与重启验证

所有整改任务数据写入 `data/tasks.json`，与班次/异常/历史保持一致：
- 服务重启后列表、详情、筛选、状态、状态历史全部保留
- 导出的 CSV/JSON 字节级完全相同（`node test/verify.js` 中的 [6/8] 会对前后快照做精确比对）

## 回归脚本（覆盖所有边界）

```bash
npm test
# 或
node test/verify.js
```

脚本包含 **8 个核心阶段**：
1. **交接班主流程**：创建→提交→确认→登记异常→复核→处理关闭→关闭班次
2. **错误边界**：员工越权关闭、跨门店越权、重复确认、未关闭异常尝试关闭、交接班人相同检查
3. **整改任务**：创建→分派→提交→验收/驳回循环完整流程，含并发冲突测试
4. **CSV/JSON 导出**：班次、异常、任务的导出格式验证
5. **设备管理与维修闭环**：创建设备、CSV导入（含重复编号处理）、巡检模板、巡检单、异常转维修、维修全生命周期
6. **权限测试**：海淀店店长只能查看本店数据、导出权限与列表过滤一致、员工跨店访问拒绝
7. **操作历史**：验证所有 23 种动作类型完整记录
8. **Phase 2 重启持久化**：真杀进程重启服务，逐字段比对数据一致性，包含 409 冲突测试

脚本使用独立临时数据目录，不会污染项目的 `data/` 目录。

## 本地配置

- `config/stores.json`：门店信息，可新增/修改/删除门店
- `config/checklist.json`：常用交接清单项，支持必填/可选
- `config/users.json`：用户账号（密码明文，仅本地演示用）
- `data/shifts.json`：班次数据（运行期生成）
- `data/exceptions.json`：异常数据
- `data/history.json`：操作历史
- `data/tasks.json`：整改任务数据

修改 `config/` 下的文件后重启服务生效。
